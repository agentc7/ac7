# Agents, slots, and authority

ac7 doesn't have a concept of "an agent" as a database row.
It has **slots**. A slot is a named seat in a team with a
name, a role, an authority level, and a bearer token. Anything
that holds the token and speaks the protocol is that slot — for as
long as it holds the token.

This matters because agents come and go. A Claude session crashes
and restarts; a container spins up; an individual contributor moves between
machines. The slot survives all of it. The identity is the slot,
not the process.

## Names

The name is the human-readable handle individual contributors address. It's
short, unique within a team, and case-insensitive:

```bash
ac7 push --agent scout --body "review the PR"
ac7 push --agent builder --body "run the migration dry-run"
```

Names are chosen at team creation time and can be anything —
functional names (`scout`, `builder`, `watcher`), numbered slots
(`agent-1`, `agent-2`), or whatever matches how your team thinks.
The broker doesn't care. The first-run wizard defaults to
`individual contributor-1` for the first slot, but you can change it to anything.

## Roles

A role is a key into the team's roles map. Each role has a
free-form `description` and a free-form `instructions` string, plus
an optional `editor` flag that lets the slot enroll in TOTP for web
UI access:

```json
{
  "roles": {
    "director": {
      "description": "Leads the team.",
      "instructions": "Set direction, assign objectives, review traces.",
      "editor": true
    },
    "implementer": {
      "description": "Writes and ships code.",
      "instructions": "Own assigned objectives end-to-end; post status updates."
    }
  }
}
```

Roles show up in `ac7 roster` output and in the web dashboard so
individual contributors can tell who does what at a glance. They also appear in
the slot's **briefing** — the composed context string the runner
injects as MCP `instructions` at session start. They don't grant
permissions; that's what authority is for.

## Authority

This is the real access control primitive. Every slot has one of
three authority levels:

| Authority | What it permits |
|---|---|
| **director** | Full team power: create/assign objectives, reassign, cancel any objective, manage watchers, **view captured LLM traces** |
| **manager** | Create objectives (becomes the `originator`), cancel objectives they originated, post to discussion threads they're members of |
| **individual contributor** | Execute assigned work: update own status, complete assigned objectives, post to threads they're members of |

Authority is set in the team config and checked server-side on
every mutating endpoint. A slot cannot elevate its own authority at
runtime. If you want a slot to become a director, edit the config
and restart the broker.

Some practical implications:

- A director + manager can both create objectives, but only a
  director can **reassign** one.
- The assignee of an objective is always the one who calls
  `objectives_complete` — not the director. Even directors can't
  complete work they haven't been assigned.
- Activity uploads (`POST /agents/:name/activity`) require the
  caller to be the slot itself. Trace viewing
  (`GET /agents/:name/activity`) is director-only.

## Tokens

Each slot has a bearer token generated at team setup. The token
is:

- Printed once during the wizard and stored in `./ac7.json`
- **Rehashed to SHA-256 on first boot** — the on-disk file is
  atomically rewritten at `0o600` so the broker holds only the hash
- Re-rotatable by editing the config and restarting

Treat tokens like SSH keys. If one leaks, rotate it and redistribute.
The broker validates against the hashed version on every request,
so any process still holding the stale token starts failing
immediately.

## How slots relate to objectives

Every objective has two slot references: an **originator** (who
created it) and an **assignee** (who owns execution). The assignee
can change via reassignment; the originator never does. A discussion
thread member list is derived from both plus any explicit watchers
plus all directors.

A slot running `ac7 claude-code` will see:

- Its open objectives in the briefing's initial state
- Lifecycle events for objectives it's assigned or originates
- Tool description refreshes via `notifications/tools/list_changed`
  when the open set changes
- New discussion posts on `obj:<id>` threads it's a member of

Non-assignee slots never see traces or tool_use-level detail of
another slot's work, even if they're watchers on the objective —
that's director-only territory.
